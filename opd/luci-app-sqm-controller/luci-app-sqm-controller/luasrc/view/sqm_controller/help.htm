系统概览

当前系统由 SQM 队列控制、下载重定向到 ifb0、防火墙分类打标、tc 按 mark 分流、策略中心和日志导出组成。 下载方向依赖 ingress -> ifb0，上传方向直接在 WAN 接口整形。

01

基础整形

根据 basic_config 里的接口、上传/下载带宽和队列算法生成 HTB + fq_codel/cake。

作用：提供上传/下载方向的基础队列控制。
02

下载重定向

把 WAN ingress 流量重定向到 ifb0，让下载方向也能被 tc 分类和整形。

备注：没有 ifb0 或 ingress redirect，下载分类不会成立。
03

分类打标

当前版本主要依赖 classification 和 class_rule，通过端口规则写入 mark，再映射到类。

备注：这是“端口级粗分类”，不是厂商级应用识别。
04

策略中心

负责执行策略、读取状态快照、预览最近记录并导出 JSON/CSV。

作用：把“策略引擎”和“策略报告”统一成一个入口。

页面导览

基础设置

配置接口、带宽、队列算法、启停状态。

作用：这是最核心的运行配置页。
配置向导

按步骤写入基础配置并立即应用。

备注：适合首次部署或现场演示。

查看 service、qdisc、ECN 等当前状态。

作用：判断 SQM 是否真的挂上了。

查看带宽、延迟、丢包等实时监控结果。

备注：偏运行态观察，不负责分类细节。
分类流量统计

查看 other / gaming / streaming / bulk 四类流量，支持切换下载/上传视角。

作用：同时看 tc 类速率、规则首次命中统计和链路诊断。

快速上手

  1. 先确认 WAN 接口和带宽 在“基础设置”或“配置向导”里填好实际 WAN 接口、上传/下载带宽和队列算法。
    作用：没有正确接口和带宽，后面的所有分类与策略都不可信。
  2. 启用服务并检查 qdisc 启用后先去“状态监控”确认 1: 和 2: root qdisc 是否存在。
    备注：看页面不如直接看 qdisc，排错更快。
  3. 应用分类器并检查 ifb0 应用分类器后确认 ifb0 上有下载类和下载 filters。
    作用：下载分类成立的前提是 ifb0、redirect 和 fw filter 全部就位。
  4. 查看分类流量统计 下载方向看 ifb0，上传方向看“接口”。规则命中表是首次命中统计，不是整条流量累计统计。
    备注：现代业务大量走 443，不可能只靠端口做到精确厂商识别。
  5. 进入策略中心 先点“执行一次策略”，再看当前模式、判定原因、动作数量和最近记录预览。
    作用：验证策略逻辑和导出能力是否正常。
  6. 最后再看日志 失败时先看日志，再决定是配置问题、环境问题还是规则覆盖率不足。
    备注：日志比页面告警更接近真实故障点。

建议

首次调试时先用一条明确的 HTTP 或 HTTPS 受控流量验证分类链路，不要直接混杂浏览器流量。

注意

当前默认规则属于端口级保守覆盖。国内视频和音乐业务主要仍由 tcp/udp 443 规则承接。

排障思路

先确认链路是否通，再讨论规则是否覆盖。如果链路没通，继续加规则没有意义。

分类规则

当前版本的分类器以 proto + dport + sport 为主，先在防火墙侧打 mark，再由 tc 把 mark 映射到类。

other 默认类，所有未命中的流量都会落到这里。
gaming 面向低延迟业务，当前默认规则主要覆盖部分常见游戏 UDP 端口。
streaming 面向流媒体业务，当前默认规则覆盖 443、1935、554、8554、1755 等保守媒体端口。
bulk 面向批量下载和传输，当前默认规则覆盖 80、BT 常见端口段、51413 等。
作用：通过 mark 把不同业务导入不同 tc 类。
备注：规则命中统计页面显示的是“规则首次命中”，因为后续同连接的流量主要依赖 conntrack mark 继承，不会持续累加到分类规则计数器。
gaming: 3478, 3659, 3074, 3479-3480, 27000-27200 streaming: 443, 1935, 554, 8554, 1755 bulk: 80, 6881-6999, 51413

策略中心

策略中心已经合并了旧“策略引擎”和“策略报告”。当前页同时承载策略执行、状态快照、最近记录预览和 JSON/CSV 导出。

执行一次策略 调用 policy_once，立即运行一次策略判断并返回结果。
状态快照 读取 /tmp/sqm_policy_state.json，展示当前模式和最近调整时间。
最近记录预览 读取报告导出接口中的最近记录，方便快速判断策略是否真正生效。
导出 JSON/CSV 把策略记录导出为结构化数据，方便留档、分析或答辩展示。
作用：把策略执行和策略记录合并成一个统一入口。
备注：旧 /report 路由仍兼容，但会直接落到策略中心。

关键路径

/etc/config/sqm_controller 主配置文件，包含基础配置、分类规则和策略配置。
/usr/lib/sqm-controller/main.py 主 CLI 入口，LuCI 绝大多数按钮最终都会调用它。
/var/log/sqm_controller.log 运行日志，排查 enable、classifier、policy 失败时优先查看。
/var/log/sqm_policy.jsonl 策略记录日志，导出报告和预览最近记录都依赖它。
/tmp/sqm_policy_state.json 策略当前状态快照，策略中心会直接读取。
/tmp/sqm_classifier_rule_state.json 分类规则窗口状态缓存，用于计算首次命中的窗口增量。
作用：快速定位配置、日志、快照和临时状态文件。

常用命令

下面这些命令覆盖配置检查、运行状态、分类链路、策略中心和日志排查。每条命令都标明“作用”和“备注”，方便直接照着执行。

基础运行

/etc/init.d/sqm-controller start 作用：启动 SQM 服务并应用当前整形配置。
备注：统一服务启动入口，Web、CLI 和开机自启都应走这条路径。
/etc/init.d/sqm-controller stop 作用：停止 SQM 服务并清理当前 tc 规则。
备注：统一服务停止入口，同时会处理策略定时任务清理。
/etc/init.d/sqm-controller restart 作用：重启 SQM 服务并重新应用当前配置。
备注：修改带宽、队列算法或分类规则后优先使用这一条。
python3 /usr/lib/sqm-controller/main.py --status-json 作用：查看当前 service、qdisc 和 ECN 状态。
备注：这是 LuCI 状态页底层使用的状态接口之一。
uci show sqm_controller 作用：查看当前 UCI 配置是否已经写入。
备注：确认配置保存成功，比只看前端输入框更可靠。

分类器与统计

python3 /usr/lib/sqm-controller/main.py --apply-classifier 作用：重新生成防火墙分类规则和 tc 分类映射。
备注：每次修改 classification 或 class_rule 后都应执行一次。
python3 /usr/lib/sqm-controller/main.py --clear-classifier 作用：清理分类器相关的防火墙规则和 tc 过滤器。
备注：分类链路异常时，可先清再重新应用。
python3 /usr/lib/sqm-controller/main.py --get-class-stats --dev ifb0 作用：查看下载方向的基础分类统计。
备注：只关注下载方向时，先看它是否有类速率变化。
python3 /usr/lib/sqm-controller/main.py --get-classifier-state --dev ifb0 作用：查看下载方向完整聚合状态。
备注：包含 summary、categories、rules、pipeline 和 diagnostics。
python3 /usr/lib/sqm-controller/main.py --get-classifier-state --dev iface 作用：查看上传方向完整聚合状态。
备注：页面里的“接口（上传）”就是这个视角。

tc / ifb0 / nft 排查

tc qdisc show dev eth1 作用：检查上传方向 root qdisc 是否已挂上。
备注：正常应能看到 1: root。
tc qdisc show dev ifb0 作用：检查下载方向 qdisc 是否已挂到 ifb0。
备注：正常应能看到 2: root。
tc class show dev ifb0 作用：查看下载类是否已经创建。
备注：应能看到 2:20 / 2:21 / 2:22 / 2:23。
tc filter show dev eth1 parent ffff: 作用：检查 ingress redirect 和 mark restore 链路。
备注：当前环境正常应出现 connmark 和 mirred redirect dev ifb0。
tc filter show dev ifb0 parent 2: 作用：检查下载方向 fw filter 是否映射到类。
备注：正常应看到 2:21 / 2:22 / 2:23 对应的 filter。
nft list chain inet sqm_fw sqm_classify 作用：查看当前 nft 分类规则和命中计数。
备注：这是判断规则是否实际命中的核心命令。

策略中心

python3 /usr/lib/sqm-controller/main.py --policy-once 作用：立即执行一次策略判断。
备注：调试策略逻辑时最常用，不必等周期调度。
python3 /usr/lib/sqm-controller/main.py --export-report --format json 作用：导出策略记录 JSON。
备注：适合做结构化分析或在浏览器中直接查看。
python3 /usr/lib/sqm-controller/main.py --export-report --format csv 作用：导出策略记录 CSV。
备注：适合导入表格工具或答辩材料。
cat /tmp/sqm_policy_state.json 作用：查看当前策略状态快照。
备注：策略中心页面读取的就是这个状态文件。
tail -n 20 /var/log/sqm_policy.jsonl 作用：查看最近策略记录。
备注：比页面预览更原始，适合确认导出内容是否正常。

日志与界面

tail -n 100 /var/log/sqm_controller.log 作用：查看应用日志的最近 100 行。
备注：排查后端异常时优先看它。
logread | grep -i sqm 作用：从系统日志中筛 SQM 相关消息。
备注：适合快速确认服务是否报错或重启。
logread | grep -E 'classifier|policy|ifb0|connmark' 作用：聚焦分类器、策略和下载链路相关日志。
备注：链路问题一般都能在这里看到关键词。
/etc/init.d/uhttpd restart 作用：重启 LuCI Web 服务。
备注：页面改动或缓存异常时再用，不是优先排障手段。

排障说明

先区分是链路没通，还是规则没覆盖。若 pipeline 里 firewall / mark_restore / tc 都是正常，那通常是规则覆盖率不足，大量流量没命中现有规则。

排障重点：先看 nft list chain 和 tc -s class show dev ifb0。

因为当前表格统计的是“规则首次命中”。同连接后续流量大多继承 conntrack mark，不会继续累加到分类规则计数器。

备注：真正的持续吞吐看 tc 类统计，不要只盯规则表。

因为当前实现主要是端口规则，现代国内视频和音乐业务大多跑在 HTTPS / QUIC 443 上。当前版本只能做保守覆盖，做不到按厂商精确识别。

作用：这是当前实现边界，不是配置写错。

因为当前功能规模下，策略执行、状态快照、预览和导出分成两个页面没有必要。现在统一到“策略中心”，旧 /report 路由只保留兼容跳转。

备注：这样更简洁，也更符合当前项目体量。

/etc/config/sqm_controller 已被声明为 conffile。升级时 opkg 会保留设备上的现有配置，不会自动覆盖。

排障重点：默认配置更新后，需要手工同步到路由器现有配置。
